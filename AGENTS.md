# AGENTS.md — adrianoalmeida.dev

This repository is a Next.js website that also contains an automated editorial pipeline for publishing a daily tech news digest.

## Repository purpose

The project is not only a personal site/blog. It also runs a scheduled content pipeline that:
- fetches tech news from RSS feeds
- ranks and filters items
- generates Markdown for a daily digest
- opens an automated pull request with draft + published files
- stores the digest metadata in DynamoDB

This automation is an important operational part of the repository and must be understood before making changes.

## Core architecture

### Frontend / site
- Next.js site
- blog content stored as Markdown in the repository
- public blog pages rendered from files in `content/blog/published`

### Editorial automation
- Vercel Cron triggers the daily digest route
- the route runs a serverless editorial worker
- the worker fetches RSS content, scores items, generates Markdown, opens a GitHub PR, and stores a digest record in DynamoDB

## Canonical blog content locations

- `content/blog/published/*.md` → public posts
- `content/blog/drafts/*.md` → draft posts generated by the editorial pipeline

When the automation runs successfully, it writes both a draft and a published version into the repository on an automation branch, then opens a PR against the base branch.

## Daily digest pipeline: where it starts

### Vercel Cron schedule
File:
- `vercel.json`

Current cron path:
- `/api/cron/news-digest`

The cron schedule is configured in Vercel and calls this route automatically every day.

### API entrypoint
File:
- `src/pages/api/cron/news-digest.ts`

Responsibilities:
- allow only `GET` and `POST`
- validate authorization via `CRON_SECRET`
- load runtime configuration from env
- call `runNewsDigest(config, mode)`
- support dry-run mode through query params

## Daily digest pipeline: orchestration

File:
- `src/lib/news-digest/index.ts`

High-level flow:
1. collect items from RSS feeds
2. rank and filter items using editorial scoring
3. generate digest content in Markdown
4. compute output paths for draft + published blog files
5. if dry-run: return generated content without writing externally
6. if live mode:
   - create/update an automation branch
   - write digest files into the repository
   - open or reuse an open pull request
   - persist digest metadata in DynamoDB

### Branch naming
Current automation branch naming pattern:
- `codex/news-digest-<slug>`

This matters because downstream tooling or manual review may rely on that branch shape.

## GitHub automation details

File:
- `src/lib/news-digest/github.ts`

This file contains the repository-writing behavior.

It is responsible for:
- checking whether the target branch already exists
- creating the branch from the base branch if needed
- creating or updating the draft Markdown file
- creating or updating the published Markdown file
- opening a pull request if one is not already open for that branch

### Important implementation detail
This repository uses the GitHub REST API directly, not the local git CLI, for editorial PR automation.

That means changes to PR behavior, branch naming, file paths, commit messages, or PR body/title should usually be made in `src/lib/news-digest/github.ts` and/or `src/lib/news-digest/index.ts`.

### Current PR shape
Title format:
- `Daily digest: <digest title>`

PR body includes:
- note that it is an automated PR
- selected item count
- draft path
- published path
- reminder to review editorial context/links/tone before merge

## Content generation and editorial logic

Files:
- `src/lib/news-digest/sources/rss.ts`
- `src/lib/news-digest/editorial.ts`
- `src/lib/news-digest/markdown.ts`
- `src/lib/news-digest/types.ts`

Responsibilities split:
- `sources/rss.ts` → fetch RSS items from configured sources
- `editorial.ts` → score, rank, deduplicate, and filter items
- `markdown.ts` → generate digest Markdown content
- `types.ts` → shared types for the editorial pipeline

When changing content quality, tone, ranking rules, or number of selected items, start here.

## DynamoDB persistence

File:
- `src/lib/news-digest/dynamodb.ts`

After the PR is opened, the pipeline stores a digest record in DynamoDB.

Default table name:
- `abolhatech-daily-news`

Stored metadata includes:
- slug
- title
- excerpt
- published markdown
- draft/published repository paths
- branch name
- PR URL
- selected item list
- creation timestamp

### Important operational meaning
This table is not just generic storage. It is part of the editorial workflow state and may now also be part of downstream processing in the broader AbolhaTech ecosystem.

## Behavior summary for future agents

When someone says:
- “a PR is opened automatically every day”
- “the Vercel cron publishes a new news item”
- “the blog digest is generated daily”
- “the cron creates content and opens a PR”

They are referring to this exact pipeline:

1. `vercel.json` schedules `/api/cron/news-digest`
2. `src/pages/api/cron/news-digest.ts` authenticates and invokes the pipeline
3. `src/lib/news-digest/index.ts` orchestrates collection, ranking, generation, PR creation, and storage
4. `src/lib/news-digest/github.ts` creates/updates files and opens the PR
5. `src/lib/news-digest/dynamodb.ts` persists digest metadata

## Safe modification guidance

When editing this automation, avoid changing multiple independent concerns at once.

Preferred approach:
- change scheduling separately from content generation
- change editorial ranking separately from GitHub PR behavior
- change storage schema carefully and intentionally
- preserve dry-run mode whenever possible

## Common future tasks and where to edit

### Change cron timing
- `vercel.json`

### Change auth behavior for cron
- `src/pages/api/cron/news-digest.ts`

### Change source feeds
- `src/lib/news-digest/config.ts`
- `src/lib/news-digest/sources/rss.ts`

### Change editorial filtering/scoring
- `src/lib/news-digest/editorial.ts`

### Change generated Markdown structure or tone
- `src/lib/news-digest/markdown.ts`

### Change PR title/body/branch/commit behavior
- `src/lib/news-digest/github.ts`
- `src/lib/news-digest/index.ts`

### Change what is saved in DynamoDB
- `src/lib/news-digest/dynamodb.ts`
- possibly `src/lib/news-digest/types.ts`

## Validation guidance for AI agents

If changing this pipeline, prefer validating with:
- local dry-run against `/api/cron/news-digest?dryRun=1`
- review of generated Markdown
- review of GitHub API assumptions
- build/lint checks if available

Do not claim the cron ran in production unless it was actually triggered and observed.
Do not claim a PR was opened unless GitHub confirmed it.
Do not claim DynamoDB writes succeeded unless the runtime actually executed them.
